2. Como funciona hoje

Modelo atual: página estática no navegador

Quando a aplicação está apenas em HTML, CSS e JavaScript, grande parte da lógica fica no navegador do usuário. Isso é bom para protótipos, mas limitado para uso corporativo com dados sensíveis.

Arquivo HTMLPágina publicada ou enviada para acesso
NavegadorUsuário abre a página
JavaScriptExecuta filtros, botões e regras
Arquivo ou ExcelDados entram manualmente
ResultadoPainel funciona, mas sem camada segura real

Exemplo: GitHub Pages é ótimo para publicar páginas estáticas, mas não substitui backend, banco de dados, autenticação Microsoft e gestão de permissões.

3. Limitações e riscos

O que não deve ficar somente no HTML

Código de frontend pode ser visualizado, copiado ou alterado por pessoas com conhecimento técnico. Por isso, dados e regras críticas precisam ficar protegidos em servidor, API e banco de dados.

Senha no HTML

Usuário e senha dentro do HTML não são segurança real. O ideal é autenticar com a conta corporativa Microsoft.

Dados sensíveis

Informações comerciais, rankings, clientes ou regras internas não devem ficar expostas no arquivo.

Sem histórico

Alterações feitas na tela podem se perder ou não gerar rastreabilidade.

Sem acesso Microsoft integrado

Sem integração com a conta Microsoft, fica difícil controlar acesso por grupos como Admin, Gestores, Diretoria e Visualizadores.

Sem logs

Fica difícil saber quem acessou, quem alterou e quando alterou.

Sem gestão centralizada

O ideal é o TI liberar ou remover acessos usando grupos da Microsoft, conforme área e função.

4. Modelo recomendado

Aplicação com login Microsoft, API e banco de dados

A evolução ideal é manter a tela simples e bonita, mas conectar a aplicação a uma camada segura controlada pelo TI.

UsuárioAcessa a aplicação
Login MicrosoftConta corporativa validada pelo TI
FrontendHTML, CSS e JavaScript
API ou backendRegras, validações e permissões
Banco de dadosDados, histórico, logs e backup
O HTML deve ser a interface. O acesso deve usar a conta Microsoft da empresa, e os dados precisam estar em uma camada controlada pelo TI.

Sugestão técnica: avaliar autenticação via Microsoft Entra ID, aproveitando as contas corporativas já existentes e controlando permissões por grupos.

5. O que precisamos do TI

Solicitações objetivas

A proposta não é substituir o TI. É permitir que o Setor de Inteligência Estratégica desenvolva protótipos e evolua os que forem úteis para um padrão seguro da empresa.

Definir ambiente seguro para hospedar as aplicações internas.
Usar conta corporativa Microsoft para autenticação, evitando senha própria no HTML.
Criar ou indicar banco de dados adequado para os painéis.
Implementar login com conta corporativa Microsoft, preferencialmente via Microsoft Entra ID.
Definir grupos e perfis de acesso na Microsoft, como administrador, gestor, diretoria e visualizador.
Criar API para o frontend buscar, gravar e atualizar dados.
Garantir backup, logs de acesso e histórico de alterações.
Definir o que pode ficar no GitHub Pages e o que precisa ir para ambiente interno.
Orientar padrões mínimos de segurança antes de publicar novos painéis.

A inteligência artificial deve ser utilizada apenas como ferramenta de apoio. Jamais, em hipótese alguma, deve substituir o conhecimento técnico, a análise humana, as regras de segurança ou a governança definida pelo TI.

7. Prioridades

Primeiros passos sugeridos

Para começar de forma prática, podemos priorizar uma aplicação piloto e definir um padrão para os próximos painéis.

Fase 1 · Diagnóstico

Mapear quais painéis existem, quais dados usam, quem acessa e quais riscos precisam ser tratados.

Fase 2 · Padrão técnico

Definir ambiente, banco, login Microsoft, autenticação, API, uso responsável de IA como apoio e regras mínimas de publicação.

Fase 3 · Projeto piloto

Escolher um painel para transformar em aplicação segura com banco e login corporativo.

Fase 4 · Escala

Reaplicar o padrão nos próximos projetos e documentar o fluxo de desenvolvimento.

Resultado esperado: mais velocidade para o negócio, com mais segurança para a empresa.

8. Mensagem final

O pedido é parceria com o TI

O Setor de Inteligência Estratégica consegue criar telas, lógica visual, filtros, rankings e protótipos úteis para o negócio. Mas, para dados corporativos, é necessário evoluir para uma estrutura com segurança, banco de dados, login Microsoft e governança.

O que o Setor de Inteligência Estratégica continua fazendo

Protótipos, layout, experiência do usuário, regras de visualização, dashboards em HTML, estudo técnico, uso de IA como apoio e validação com a área de negócio.

O que precisamos do TI

Backend, banco de dados, autenticação Microsoft, permissões por grupo, ambiente seguro, logs, backup e padrões de publicação.

A inteligência artificial soma ao conhecimento da equipe, mas jamais substitui pessoas, critérios técnicos, segurança ou governança. A evolução para aplicações internas seguras precisa ser feita em parceria com o TI.
